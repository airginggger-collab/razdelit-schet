# Разделить счёт — дизайн-спек

**Дата:** 2026-06-24  
**Статус:** утверждён

---

## Что строим

PWA-приложение для дележа счёта в кафе/ресторане. Один человек (организатор) открывает приложение, сканирует или вводит счёт, добавляет участников, делит — и видит кто уже закинул деньги через анимированные полоски. Ссылкой можно поделиться с остальными.

---

## Платформа и стек

- **PWA** — устанавливается на экран телефона, работает офлайн
- **Vanilla JS + HTML + CSS** — без фреймворков, один билд не нужен
- **Tesseract.js** — OCR в браузере, грузится лениво (~4 МБ) только при выборе сканирования
- **Service Worker** — кэширует всё для офлайн-работы
- **Без сервера** — шеринг через URL hash (JSON → Base64), история в localStorage

## Экраны (4 шага)

Переходы анимированы: slide влево/вправо. Кнопка «назад» на экранах 2–3 (на экране 1 её нет — возвращаться некуда).

### Экран 1 — Добавить счёт

Два режима на выбор:

**Фото чека** — открывает камеру, фотографирует, Tesseract.js распознаёт текст, парсит позиции и суммы. Результат всегда редактируемый (список позиций можно поправить руками).

**Ввести вручную** — поле «Итого» (число) и опционально список позиций (название + сумма). Можно просто ввести одну общую сумму без позиций.

CTA: красная кнопка «Далее».

### Экран 2 — Участники

- Счётчик «Нас» (−/+), минимум 2
- Опционально: добавить имена (поле с аватаром-инициалом)
- Без имён — участники называются «Участник 1», «Участник 2» и т.д.
- Если позиции были введены на экране 1 — можно назначить кто что ел (чекбоксы у позиций). Если позиций нет — делит поровну.

CTA: «Далее».

### Экран 3 — Итог

- Большое число «Каждый платит: X ₸» по центру (при равном делении)
- Если назначены позиции — индивидуальные суммы для каждого
- Список позиций счёта
- Подпись: «из N ₸ · M человек»

CTA: «Начать сбор» → переход на Экран 4.

### Экран 4 — Трекер (главный)

- Общая полоска вверху: «Собрано X / N ₸» + анимированное заполнение
- Карточка каждого участника:
  - Аватар с инициалом (цвет уникальный на участника)
  - Имя
  - Сумма к оплате
  - Индивидуальная полоска (0% → 100%)
  - Статус: ожидает / закинул ✓
- **Тап по карточке** = отметить «закинул» (полоска заполняется с анимацией)
- **Ввод суммы** = долгое нажатие или иконка-карандаш → поле ввода (частичная/другая сумма)
- **Кнопка «Поделиться»** (иконка share вверху справа) — генерирует URL hash, открывает системный share sheet

Когда все закинули — конфетти-анимация + «Готово!».

## Данные

```js
// session (localStorage + URL hash)
{
  total: 12400,
  currency: '₸',
  items: [{ name: 'Шашлык', price: 4200 }],  // может быть пустым
  participants: [
    { id, name, color, share, paid, paidAmount }
  ]
}
```

**localStorage** — хранит последние 10 сессий (история).  
**URL hash** — Base64(JSON.stringify(session)), обновляется при каждом тапе «закинул».

**Округление:** доли участников квантуются в целые ₸ методом наибольшего остатка (`roundShares`), чтобы сумма выводимых сумм точно равнялась счёту (100 ₸ на троих → 33+33+34, а не 33+33+33=99). Русское склонение «человек/человека» — через `pluralPeople`.

## Шеринг без сервера

Состояние сессии кодируется в URL hash. Когда организатор тапает «Поделиться» — в буфер и в share sheet уходит текущий URL. Остальные открывают ссылку и видят то же состояние. Каждый может тапнуть свою карточку — но URL обновится только у него (нет реального real-time sync). Для реального сбора организатор — единственный, кто ведёт трекер.

Это сознательное ограничение MVP: real-time sync потребует сервер.

## Визуальный стиль

- Референс: Kaspi Bank — белые карточки, красный акцент (#E8131E), чистые числа
- Типографика: system-ui, числа крупные и жирные
- Анимации: CSS transitions 300ms ease, полоски — spring через JS (requestAnimationFrame)
- Конфетти при 100% сборе: canvas-based, 1.5 сек
- Без тёмной темы в MVP

---

## OCR

Tesseract.js загружается только при выборе «Сфоткать чек». Парсим: ищем строки вида `Название ... 1 234` или `1 234.00`. Общий итог — строка со словом «итого»/«total»/«сумма». Все результаты редактируемые — пользователь может поправить любую позицию до перехода дальше.

Парсер консервативен: фильтрует не-товарные строки (ИНН/БИН, телефоны, даты, номера чеков), отделяет количество (`x2`) от цены и отбрасывает позиции дороже всего счёта. Распознанный «Итого» имеет приоритет над суммой позиций (учитывает налог/сервис) и не перезатирается ею.
