# Система работы над проектом — пошаговая методология

> Единый источник. Сведено из разрозненных версий по проектам: portfolio `/process` + `docs/process.md`, karaokeshop `docs/kak-ya-rabotayu.md/html`, medlog `docs/HOW_I_WORK.md` / `public/how-i-work.html`, `docs/project-kickoff-script.md`. Снимок на 2026-06-24.
> Принципы выведены из реального опыта **MedLog · karaokeshop · WhatDaDev**.

---

## 0. Принципы, зашитые во все фазы

Это не «советы», а правила, нарушение которых уже стоило времени на прошлых проектах.

| Принцип | Что значит | Откуда урок |
|---|---|---|
| **Гейты между фазами** | Нельзя переходить к следующей фазе без письменного OK клиента. Устное «вроде ок» — не гейт. | Все проекты |
| **Решения до кода (Decision Log / ADR)** | Каждое нетривиальное решение фиксируется письменно: контекст → варианты → выбор → аргумент. | karaokeshop (ADR-0001/0002) |
| **Аналитика в первый коммит** | Счётчик/Umami ставится сразу, не «потом». Иначе нечем мерить, что переписывать. | WhatDaDev (редизайн вслепую без данных) |
| **Деплой в первый день** | Рабочая заготовка деплоится в день 1. Прод существует с начала, а не в конце. | Все проекты |
| **Один источник правды на тип данных** | Цвет — в одном файле. Контент — в одном файле. Дубль = тихий рассинхрон. | karaokeshop (tokens.json ↔ globals.css конфликт) |
| **Figma врёт — тестируй на устройстве** | Мокап ≠ реальный рендер. Проверка на реальном телефоне, не в дизайне. | MedLog (mobile-first) |
| **Максимум 2 раунда правок по дизайну** | Дальше — вне scope = новый бриф. Защищает и сроки, и отношения. | Все проекты |
| **Доки в том же коммите** | CHANGELOG/HANDOFF/контекст обновляются вместе с кодом, не отдельным заходом. | MedLog (строгое правило) |

## Фаза 0 — Интервью и брифинг

**Цель:** понять бизнес и ограничения раньше, чем открыт редактор.

**Шаги — 4 блока вопросов:**
1. **Бизнес** — что продаёт/делает, кому, чем зарабатывает, кто конкуренты, в чём отличие.
2. **Сайт** — зачем нужен (продажи / лиды / бренд / SEO), что есть сейчас, что болит.
3. **Контент** — что уже есть (тексты, фото, прайс), что нужно создать, кто даёт материалы.
4. **Технические ограничения** — бюджет на инфраструктуру, резидентность данных, домены/доступы, интеграции (CRM, оплата), кто будет вести сайт после.

**Артефакты:** заметки интервью, список ограничений, список того, что ждём от клиента.
**Гейт:** клиент подтвердил, что цели и ограничения зафиксированы верно.
**Ловушка:** не начинать дизайн без ответа на «зачем сайт» и «кто ведёт после». Без этого переписываешь вслепую.

## Фаза 1 — Структура и воронки (IA)

**Цель:** карта страниц от пользовательских сценариев, не от «хочу красиво».

**Шаги:**
1. Выделить **воронки** (например B2C и B2B — разные потребности → разные пути).
2. Из воронок → **карта страниц** (sitemap): что есть у каждого сценария.
3. Расставить **приоритеты** страниц (что в MVP, что фаза 2).
4. Для SEO-проектов — семантическое ядро → URL-карта от него (а не наоборот).

**Артефакты:** sitemap / IA-карта, URL-карта, приоритизация.
**Гейт:** клиент согласовал структуру и слаги.
**Ловушка:** группировать по техническим сущностям («Записи/Данные») — пользователь так не думает. Группировать по сценариям («Сегодня/Визиты/Лекарства»).

## Фаза 2 — Решения до кода (ADR / Decision Log)

**Цель:** зафиксировать ключевые архитектурные решения письменно, прежде чем писать код.

**Шаги — по каждому решению:**
1. **Контекст** — что решаем и почему сейчас.
2. **Варианты** — реально рассмотренные, с плюсами/минусами.
3. **Выбор + аргумент** — почему именно так.
4. Что выбор разблокирует / какие последствия.

**Типичные решения:** цветовая система и токены, стек, хостинг/деплой, CMS vs статика, резидентность данных, модель данных, аналитика.

**Артефакты:** `docs/adr/NNNN-*.md` (по решению), сводный Decision Log.
**Гейт:** блокирующие решения, требующие ответа клиента (резидентность, CMS, домен), — отправлены клиенту одним листом и получены ответы.
**Ловушка:** «решим по ходу». Решение, принятое в коде без фиксации, через месяц никто не вспомнит, а дубль источников разойдётся.

## Фаза 3 — Визуализация и утверждение стиля

**Цель:** согласовать визуальный язык **без вёрстки** — дешевле менять токены, чем HTML/CSS.

**Шаги:**
1. Дизайн-токены: палитра (с WCAG-контрастом), типографика, радиусы, отступы — **один источник** (`theme.css` / `tokens.json`).
2. Референсы и направление (тон: минимал / премиум / и т.д.).
3. Согласование стиля на токенах/референсах, не на готовых страницах.

**Артефакты:** файл токенов, борд референсов, мини-гайд стиля.
**Гейт:** клиент утвердил визуальное направление.
**Ловушка:** показывать клиенту полу-свёрстанные страницы — правки полетят в код, дорого. И: **не дублировать цвета** в двух местах — вернётся «правлю — не меняется».

## Фаза 4 — Первый коммит и деплой (день 1)

**Цель:** рабочая заготовка в проде в первый день.

**Шаги:**
1. Скелет проекта по утверждённым решениям (стек из Фазы 2, токены из Фазы 3).
2. Ленивая загрузка страниц, барьеры модулей, дата-роутер (по стеку).
3. **Аналитика — в этот же коммит.**
4. **Деплой** — настроить авто-деплой (push в основную ветку → прод).
5. Прислать клиенту живой URL.

**Артефакты:** живой прод-URL, репозиторий, настроенный CI/деплой.
**Гейт:** заготовка открывается на проде, аналитика шлёт события.
**Ловушка:** копить локально неделю до «показа». Прод с первого дня = ранняя обратная связь и нет сюрпризов на сдаче.

## Фаза 5 — Итерации (рабочий цикл)

**Цель:** предсказуемый цикл на каждое изменение.

**Шаги цикла (на каждую задачу):**
```
задача → обсуждение → ADR (если нетривиально) → реализация
       → проверка (build + устройство) → коммит (+доки) → пуш/деплой
```

**Правила цикла:**
- Нетривиальное решение → сначала ADR, потом код.
- Проверка перед «готово»: сборка зелёная + визуал на реальном устройстве/превью.
- Доки (CHANGELOG/HANDOFF/контекст) — в том же коммите.
- Максимум 2 раунда правок по дизайну на блок; дальше — новый бриф.

**Артефакты:** коммиты с осмысленными сообщениями, обновлённый CHANGELOG/HANDOFF.
**Гейт (на каждую задачу):** проверено и задеплоено, клиент видит результат на проде.
**Ловушка:** «причешу потом» по дизайну превращается в whack-a-mole. Чинить корень (архитектуру), а не симптом.

## Фаза 6 — Проверка и передача (handoff)

**Цель:** сайт можно отдать другому разработчику/владельцу без тебя.

**Шаги — чеклист перед запуском:**
- **Техника:** сборка/тесты зелёные, нет оверфлоу/битых ссылок, 404 работает, мобайл проверен на устройстве.
- **SEO:** уникальные title/description, один H1, canonical, sitemap/robots, OG-теги, структурированные данные.
- **Контент:** реальные тексты/фото (не плейсхолдеры), контакты верные.
- **Доступы/инфра:** домен подключён, аналитика активна, секреты на месте.

**Артефакты передачи:**
- `README` / `CLAUDE.md` — стек, команды, конвенции.
- `ARCHITECTURE.md` — карта слоёв/папок.
- `docs/HANDOFF.md` — снимок состояния, что live, что дальше.
- `dev-howto` / `contributing` — как добавить страницу/компонент/модуль.
- Инструкция владельцу-новичку (как править через браузер).
- Для бэка (если планируется): контракты, content-model, env.example.

**Гейт:** новый человек по докам поднимает проект и делает первый PR без тебя.
**Ловушка:** доки «потом». Передавать нужно так, будто ты завтра исчезаешь.

## 8 правил, которые экономят время на следующем проекте

1. **Гейт = письменный OK.** Нет письма — нет перехода к следующей фазе.
2. **Decision Log с первого дня.** Решение без аргумента = будущий спор.
3. **Аналитика в первый коммит.** Без данных редизайн — это гадание.
4. **Деплой в день 1.** Прод существует с начала.
5. **Один источник правды на тип данных.** Цвет/контент — в одном месте.
6. **Тестируй на устройстве, не в Figma.**
7. **Доки в том же коммите, что и код.**
8. **2 раунда правок — потолок.** Дальше — новый scope/бриф.
